# HIMS Frontend — Cloudflare Deployment Guide (Cloud / Production)

**Scope note:** This guide covers ONE deployment target — Cloudflare Pages —
as requested. The application code itself is 100% platform-independent:
it is a standard Vite + React static build with no Cloudflare-specific code
anywhere in `src/`. Nothing in this zip locks you into Cloudflare. You can
run `npm run build` and upload the resulting `dist/` folder to Netlify,
Vercel, S3 + CloudFront, GitHub Pages, or any nginx server — your existing
`nginx.conf` in this same zip already covers the traditional-server path.

The only two files added specifically for convenience here — `public/_headers`
and `public/_redirects` — are a shared convention also read by Netlify, and
are silently ignored (harmless) by every other host. No file in `src/`,
`index.html`, `vite.config.js`, or `package.json` was changed.

---

## 0. What's in this zip vs. what you had

| Removed from this delivery (not from your project) | Why |
|---|---|
| `node_modules/` | Reinstalled fresh on any machine/CI with `npm ci` — never shipped |
| `dist/` | Build output — regenerated by the deploy pipeline itself |
| `certs/localhost.pem`, `certs/localhost-key.pem` | Local dev-only self-signed cert; meaningless off your laptop. Cloudflare terminates TLS for you automatically |
| `.env.local` | Your local machine's `VITE_API_URL=https://localhost:8001` — not a production value |

| Added for this delivery | Why |
|---|---|
| `public/_headers` | Cloudflare/Netlify convention file — replicates the exact security header set from your `nginx.conf` (HSTS, CSP, X-Frame-Options, etc.) so the security posture doesn't regress when you move off nginx |
| `public/_redirects` | SPA fallback rule (`/* /index.html 200`) — Cloudflare Pages equivalent of nginx's `try_files ... /index.html` |
| `.env.production.example` | Documents the one required build variable, `VITE_API_URL` — for reference only, not read by any tool |

Nothing else changed. `src/`, `index.html`, `vite.config.js`, `package.json`,
`package-lock.json` are byte-for-byte what you uploaded.

## 1. Prerequisites

- A Cloudflare account (free tier is enough to start): https://dash.cloudflare.com/sign-up
- Your HIMS backend already deployed and reachable over **https://** at a
  real public URL (e.g. `https://api.yourhospital.com`). The frontend will
  refuse to run without a valid `https://` `VITE_API_URL` — this is enforced
  in `src/api/himsApi.js` (unchanged, existing behavior).
- This project pushed to a GitHub or GitLab repository (recommended path),
  **or** Node.js 18+/npm 9+ installed locally if you prefer direct CLI upload
  (Option B below).

---

## 2. Option A — Git-connected deploy (recommended)

This is the standard, ongoing-deployment path: every push to your repo
triggers an automatic rebuild.

### 2.1 Push this project to GitHub/GitLab

```bash
cd hims-frontend-cloud
git init
git add .
git commit -m "HIMS frontend v2.3 — cloud deployment package"
git branch -M main
git remote add origin https://github.com/YOUR_ORG/hims-frontend.git
git push -u origin main
```

> `node_modules/`, `dist/`, and `certs/*.pem` are already excluded via the
> existing `.gitignore` in this project — no changes needed there either.

### 2.2 Create the Cloudflare Pages project

1. Go to **Cloudflare Dashboard → Workers & Pages → Create → Pages → Connect to Git**.
2. Authorize Cloudflare to access your GitHub/GitLab account and select the
   `hims-frontend` repository.
3. Choose the branch to deploy from production (e.g. `main`).

### 2.3 Build configuration

On the "Set up builds and deployments" screen, enter:

| Setting | Value |
|---|---|
| Framework preset | `Vite` (or "None" — the settings below are explicit either way) |
| Build command | `npm run build` |
| Build output directory | `dist` |
| Root directory | `/` (leave default, unless you placed this project in a subfolder) |
| Node version | `18` or higher — set via environment variable `NODE_VERSION=20` to match your CI (`.github/workflows/ci.yml` uses Node 20) |

### 2.4 Environment variables (required before first build)

Still on the setup screen (or later under **Settings → Environment variables**),
add:

| Variable name | Value | Environment |
|---|---|---|
| `VITE_API_URL` | `https://api.yourhospital.com` (your real backend URL — **must** be `https://`) | Production |
| `VITE_API_URL` | your staging backend URL, if different | Preview |
| `NODE_VERSION` | `20` | Production and Preview |

This is the single variable `himsApi.js` requires. If it's missing or not
`https://`, the app throws a clear startup error by design (existing,
unchanged behavior) rather than silently pointing at the wrong backend.

### 2.5 Deploy

Click **Save and Deploy**. Cloudflare will run `npm ci && npm run build`
(effectively — Pages uses `npm ci` when a `package-lock.json` is present,
matching your existing CI reproducibility guarantee) and publish `dist/`.

Your app will be live at `https://<project-name>.pages.dev` within a
couple of minutes.

### 2.6 Update the CSP to match your real backend domain

Open `public/_headers` in the repo and replace the placeholder domain:

```
connect-src 'self' https://api.yourhospital.com;
```

with your actual backend origin (the same value as `VITE_API_URL`). Commit
and push — Pages will redeploy automatically. Until this matches, the
browser's Content-Security-Policy will block API calls to a different
origin than what's listed here (this is intentional — same protection your
original `nginx.conf` enforced).

## 3. Option B — Direct upload via Wrangler CLI (no Git required)

Use this if you want a one-off deploy without connecting a repository.

```bash
# 1. Install dependencies
npm ci

# 2. Set the production API URL for this build only
export VITE_API_URL=https://api.yourhospital.com

# 3. Build
npm run build

# 4. Install Wrangler (Cloudflare's CLI) if you don't have it
npm install -g wrangler

# 5. Log in (opens a browser window)
wrangler login

# 6. Create the Pages project (first time only)
wrangler pages project create hims-frontend

# 7. Deploy the dist/ folder
wrangler pages deploy dist --project-name=hims-frontend
```

Repeat steps 2–3 and 7 for every subsequent deploy. `_headers` and
`_redirects` are picked up automatically from inside `dist/` (Vite copies
them there from `public/` at build time) — no extra Wrangler config needed.

---

## 4. Custom domain

1. In the Pages project, go to **Custom domains → Set up a custom domain**.
2. Enter your domain (e.g. `hims.yourhospital.com`).
3. If the domain's DNS is already on Cloudflare, the CNAME is added
   automatically. If it's on another registrar/DNS provider, add the CNAME
   record Cloudflare shows you.
4. Cloudflare issues and manages the TLS certificate automatically —
   no manual certificate handling, unlike the local/nginx path in
   `LOCAL_FRONTEND_GUIDE.md`.
5. Cloudflare enforces HTTPS by default; combined with the HSTS header in
   `_headers`, there is no plain-`http://` fallback in production, matching
   the project's locked decision (see `LOCAL_FRONTEND_GUIDE.md`, `README.md`).

## 5. Post-deploy verification checklist

- [ ] `https://<your-domain>` loads the login screen (no mixed-content or
      CSP errors in browser DevTools console)
- [ ] DevTools → Network: confirm API calls go to your real `VITE_API_URL`
      origin, over `https://`
- [ ] DevTools → Application → Storage: confirm zero `localStorage` writes,
      `sessionStorage` only (matches `storage.test.js`, unchanged)
- [ ] Refresh on a deep link (e.g. `https://<your-domain>/some/route`)
      loads correctly instead of a 404 — confirms `_redirects` is active
- [ ] Response headers on the main document include `Strict-Transport-Security`,
      `Content-Security-Policy`, `X-Frame-Options: DENY` — confirms
      `_headers` is active (check via DevTools → Network → click the
      document request → Headers)
- [ ] Login → 10-step wizard → submit → real `MR-YYYY-NNNNNN` number returned
      from your production backend
- [ ] Idle timeout (13 min warning / 15 min logout) still functions
- [ ] Forgot Password / OTP flow works against the production backend

---

## 6. Rollback

Cloudflare Pages keeps every previous deployment. To roll back:
**Workers & Pages → your project → Deployments tab → find the last-known-good
deployment → "..." menu → Rollback to this deployment.** This is instant and
requires no rebuild.

---

## 7. Ongoing updates

- **Git-connected (Option A):** every push to the production branch
  auto-deploys. Pull requests get their own preview URL automatically
  (`VITE_API_URL` Preview value from step 2.4 is used).
- **Direct upload (Option B):** re-run steps 2–3 and 7 in section 3.

Nothing about this workflow requires touching `src/` — deployment target
and the application code remain fully decoupled, so this same `dist/`
output is equally valid for any other static host if you switch providers
later.
